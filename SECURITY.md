# Security Policy

## Reporting a vulnerability

Please report suspected vulnerabilities privately, not via a public
issue or pull request.

- **Email:** contact@sebastienrousseau.com
- **Subject prefix:** `[SECURITY] euxis-publisher`
- **Include:** affected file or path, reproduction steps, and the
  impact you observed. A minimal proof-of-concept is welcome.

You will receive an acknowledgement within seven days. Do not
disclose publicly until a fix is available or thirty days have
elapsed without response.

## Scope

This repository ships the **public publishing engine**: a Python
CLI (`euxis-publisher`), LaTeX document classes and style packages
under `core/`, and a Jinja2 rendering layer. In-scope concerns:

- **Template injection** in the Jinja2 renderer (`inclusio.cli.render`).
- **Path traversal** in the build orchestrator (`inclusio.cli.build`)
  or the audit CLI (`inclusio.cli.audit`) — particularly via
  the `EUXIS_CONTENT_DIR` env var or `--content-dir` flag.
- **Command injection** in the tailor CLI's subprocess paths
  (`inclusio.cli.tailor`) or the asset pipeline shell helpers.
- **Arbitrary code execution** through unsafe YAML loading or
  unchecked LaTeX `\write18` shell-escape.
- **Supply-chain risks** introduced by `pyproject.toml` dependencies
  or by the GitHub Actions used in `.github/workflows/`.
- **PDF metadata leakage** from `tools/stamp_pdfs.py` (build-host
  paths, usernames, secrets) into camera-ready artefacts.

Out of scope:

- Vulnerabilities in upstream LaTeX engines (LuaLaTeX, latex-lab,
  tagpdf, pdfx) — report to their respective projects.
- Vulnerabilities in PyYAML, Jinja2, pikepdf, pytest — report
  upstream.
- Vulnerabilities in veraPDF or its dependencies — report to the
  veraPDF Consortium.

## Signed commits

Commits on `main` are expected to be cryptographically signed (GPG
or SSH). The PR workflow validates this on every pull request —
unsigned commits block the merge. Signing setup is documented in
[`CONTRIBUTING.md`](CONTRIBUTING.md).

## Secrets and PII

The engine itself ships no credentials. The companion private
content repository
([`euxis-publisher-private`](https://github.com/sebastienrousseau/euxis-publisher-private),
not public) holds author PII, draft patents, and tailored CV data.
Reports concerning leakage from the private repo should also be
filed via the channel above.

If you discover a credential committed by mistake — to either the
public engine or the private content repo — treat it as a
vulnerability and follow the reporting process above. Rotation
precedes disclosure.

## Supply chain

- All GitHub Actions are SHA-pinned (enforced by Dependabot under
  `.github/dependabot.yml`).
- Python dependencies are SHA-pinned in CI via `uv` and the
  generated `uv.lock`.
- Releases ship with SLSA L3 build provenance attestations
  (`.intoto.jsonl`), generated by
  [`actions/attest-build-provenance`](https://github.com/actions/attest-build-provenance)
  in `.github/workflows/release.yml`.
