## How to read this audit

Three concentric questions:

1. **Parity** — what does the private content repo do that the
   public engine should host, or expose, or test for?
2. **Catch-up** — where has the 2026 ecosystem (LaTeX team
   tagged-PDF, Quarto 1.9, Typst, MCP) moved past us between
   Sprints 0 and 3?
3. **Forward bet** — which Sprint 4-12 deliverables in
   `implementation-plan-2026.md` are now de-risked enough to start
   in parallel, versus which should stay sequenced.

Items are tagged **F0/F1/F2/F3** for forcing-function origin, with
effort (XS &lt;1d / S 1-2d / M 3-5d / L 1-2w / XL 2-4w) and
risk (Low/Med/High). Sprint references are to
`implementation-plan-2026.md`.

## 0. Headline

The public engine is in a **structurally strong but operationally
behind** position:

| | State |
|---|---|
| Tagged-PDF foundation | ✅ Sprints 0-2 done — 11/11 classes retrofitted, `\DocumentMetadata` snapshot, PDF/A-4f path live |
| Accessibility gate | ✅ Sprint 3 done — veraPDF `ua2 / wt1a / 4f` audit CLI with strict mode + JSON/MD reports |
| Operational hygiene | ⚠️ Behind the private content repo — no `CITATION.cff`, no `SECURITY.md`, no `dependabot.yml`, no release workflow, no signed-commit gate, no SHA-pinning, no `uv` in CI |
| Multi-format output | ❌ Not started — LaTeX→PDF only, no JATS/HTML/EPUB/DOCX-from-paper emitter |
| AI authoring layer | ❌ Not started — no MCP server, no Claude skill, no Cursor rules |
| LLM judges | ❌ Not started — `chktex` + `vale` are the only quality gates |
| Provenance | ⚠️ Partial — `stamp_pdfs.py` does git-hash stamping; no SLSA / Sigstore / in-toto / PAdES / C2PA |
| Patent workflow | ⏸️ Deferred (D1) — typesetting-only, by decision |
| Typst backend | ⏸️ Deferred to P2 — 24-month threat horizon |

The single most leveraged set of changes for H2-2026 is **(a) close
operational-hygiene parity with the private repo (one sprint, low
risk), (b) start the JATS/HTML emitter in Sprint 4-5 so single-source
multi-format ships before Quarto 1.10 (~Sept-2026), (c) stand up an
MCP server in Sprint 6 to make the engine visible to Claude Code /
Cursor before NISO Plus 2026 normalises that surface**.

## 1. Findings — what the private repo has that we don't

The private repo (`euxis-publisher-private`, content) shipped a
series of optimisations between 2026-05-09 and 2026-05-27 that the
public engine should mirror. None of these touch content; all are
infrastructure / hygiene / scholarly-citation surface.

| # | What private has | What public has today | Gap | Action |
|---|---|---|---|---|
| P-1 | `CITATION.cff` (CFF 1.2.0) with ORCID, repository-code, license | None | Public engine is uncitable in scholarly tooling; no "Cite this repository" button | **Add `CITATION.cff`** + `tests/test_citation.py` (parses, required fields, ORCID URI shape) |
| P-2 | `SECURITY.md` with private disclosure path, scope, signed-commits clause | None | OSS hygiene gap; no vulnerability channel published | **Add `SECURITY.md`** (engine-scoped: template injection, path traversal, supply chain) |
| P-3 | `.github/dependabot.yml` — weekly pip + GitHub Actions groups | None | GitHub Actions are unpinned and unmonitored; pip deps drift silently | **Add `dependabot.yml`** (pip + github-actions, weekly Monday, grouped) |
| P-4 | `.github/workflows/release.yml` on `v*` tag | None | No release artefact, no changelog, no SBOM, no attestation | **Add `release.yml`** with `actions/attest-build-provenance` (SLSA L3) for PDF + sdist + wheel |
| P-5 | `.github/workflows/pre-commit-autoupdate.yml` — weekly Monday | None | Pre-commit hook revisions stale; signed-PR path exists in private but not public | **Add `pre-commit-autoupdate.yml`** (sign-commits via `create-pull-request` action) |
| P-6 | All Actions SHA-pinned (`@de0fac2e…` not `@v4`) | Symbolic refs (`@v4`, `@v5`) | Supply-chain risk; Dependabot can't pin without seeing pinned form | **Migrate all CI to SHA-pinned actions** (Dependabot rewrites them after P-3) |
| P-7 | `uv` as Python installer (single binary, deterministic) | `pip install` in every workflow | CI ~30-60 s slower per job; cache miss on each run | **Switch CI to `astral-sh/setup-uv`** + cache key on `pyproject.toml` |
| P-8 | `requires-python = ">=3.12"` | `>=3.9` | 3.9 EOL'd Oct 2025; 3.10 EOL'd Oct 2026 in 5 months | **Bump to `>=3.11`** (3.11 is min-supported by `astral-sh/setup-uv`, `pikepdf>=8`, lets us drop a lot of `__future__` boilerplate) |
| P-9 | `tools/overlay.py` — `merge(base, overlay)` + `resolve(overlay, base_loader)` for "shared template, overlay data" pattern | None | Each CV variant in content repo carries a full data file fan-out (4 today, painful at 10) | **Port `overlay.py` to `inclusio/tools/overlay.py`** (pure function, no dependencies, unit-tested) so content repo can import it |
| P-10 | `tools/render_cv_text.py` — ATS-safe plain-text CV shadow | `render.py --format markdown` emits Workday markdown; no plain-text path | Some 2026 ATS pipelines (Taleo, older Workday) still extract from `.txt` better than from any PDF | **Add `render --format text`** to `cli/render.py` (single Jinja2 env, LaTeX-unescape pre-filter, write to `build/<doc>.txt`) |
| P-11 | Signed-commit gate (`git log --format='%H %G?' \| grep ' N$'`) on PRs | None | Anyone can land unsigned commits to engine `main` | **Add signed-commit job** to `engine-validation.yml` (PR only) |
| P-12 | `ruff` for Python lint + format, `target-version = "py312"` | No Python linter wired into CI | Style drift; engine code already mixes one-liners and PEP 8 formatting | **Add `ruff` to `pyproject.toml`** + a `lint-python` job in `engine-validation.yml` |
| P-13 | Pinned `pre-commit` hook revisions (`pre-commit-hooks@v6.0.0`, `ruff-pre-commit@v0.15.12`) | Pre-commit pinned but uses local-only bash hooks for chktex/latexindent | Pre-commit hooks here are functional but not auto-updated; no `ruff` hook | **Replace local bash hooks with upstream ruff + pre-commit-hooks repos** where possible; keep `chktex` / `latexindent` local (no upstream hook exists) |

Effort to clear **all** of P-1 → P-13: ~3-5 days, all Low risk,
all additive. This is the cheapest, most-visible parity win.

## 4. Findings — competitors, two-axis map

Two questions only matter for positioning: (a) who do we lose
authors to today, (b) who do we lose them to in 2027-2028.

| Competitor | Lose authors today? | Lose authors 2027-2028? | What they have that we lack |
|---|---|---|---|
| **Quarto 1.9+** | **Yes** — for scholarly + technical authors who want HTML + PDF + JATS from one source. Free, Posit-backed, accepted by JOSS / eLife / PLOS as a manuscript bundle. | Yes, harder | Single-source multi-format; `pdf-standard: PDF/UA-2` + built-in veraPDF since Mar-2026; Manuscripts feature; Pandoc backend; massive ecosystem |
| **Typst 0.x** | No — publisher acceptance ~0, math ecosystem immature for STM | **Possibly** — if a tier-1 publisher blesses it in 2027 | Accessibility on by default; UA-1 today, UA-2 likely 2026-Q4; instant compile; clean syntax; growing Rust ecosystem |
| **Pandoc-only pipelines** | Marginal — engineers comfortable with raw Pandoc | No (commodity) | Lingua franca; nothing we don't already get via Pandoc adapter |
| **Overleaf + LaTeX template galleries** | Yes for non-technical authors who never leave a browser | Yes (but Overleaf adopts whatever LaTeX team ships) | Cloud-native authoring; commenting; track-changes; institutional auth |
| **`jobsprout` / `Rezi` / `Teal`** (CV-only) | Yes for CV-only users | Yes | ATS scoring; recruiter heuristics; JD-to-CV closed loop |
| **MyST / Jupyter Book 2** | Yes for executable scientific docs | Yes | Cross-reference resolver, JATS, interactive figures, hyperlinked code |
| **ClaimMaster / PatSnap / Rowan** (patent vertical) | N/A — we deferred (D1) | N/A (deferred) | LLM-assisted drafting, prior-art, FTO — pure SaaS verticals |

The two strategic chokepoints in this table are:

1. **Quarto eats single-source HTML+PDF+JATS by Sprint 6 unless we
   ship a credible emitter.** The user base we lose first is the
   JOSS / arXiv / preprint cohort who don't want to pick between
   beautiful PDFs and accessible HTML. Forcing function: **F3**.
2. **CV-tailoring SaaS eats the JD-to-CV authoring loop unless we
   ship a closed-loop LLM judge.** This is the only segment where
   Euxis can compete on *outcomes* (interviews booked) not just
   typesetting. Forcing function: **F5**.

Everything else is a slower-burn threat.
